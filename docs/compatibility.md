# Compatibility matrix

jemmet, iotakt, henret, and kroopt are **independent projects**. jemmet depends on them;
it does not contain them. Each is consumed as a **pinned Lake git dependency** (RFC 001):
declared in `lakefile.toml` with `require … from git "<url>" @ "<rev>"`, fetched by Lake
into the gitignored `.lake/packages/`, and commit-locked by the committed
`lake-manifest.json`. No copy of a sibling's source — tree or tarball — lives in this repo.

This table records which sibling versions a given jemmet builds against. The hashes are the
peers' own published release attestations (their provenance sidecars); jemmet references
them for the record — it does not re-host them.

| jemmet | iotakt | kroopt | notes |
|--------|--------|--------|-------|
| 0.0.1-dev | **0.14.6** (published) | — (M3) | jemmet binds iotakt's henret-free **model** package (the repo root; RFC 061 split). Pinned `git "https://github.com/nabbisen/iotakt" @ "v0.14.6"`. Published release provenance: source archive `af98a8c6…` (326930 bytes), source-tree `6e2ce663…`, lake-manifest `344d7023…`, toolchain `leanprover/lean4:v4.15.0`. The native `runtime/` + henret bridge are iotakt's own out-of-tree concern, seamed at deployment via `IotaktRuntime.*` (RFC 008/009) — off jemmet's verified-core path. |

## henret (actor/task runtime — RFC 015 handler hand-off)

jemmet's RFC 015 handler-handoff binding (`JemmetHenret`) is validated against **henret
0.34.4**, pinned `git "https://github.com/nabbisen/henret" @ "v0.34.4"` =
commit `ad0ceab4ebed2884c9165be44154dca2c1f4816f` (published release provenance: archive
`ad9f0582…` / 573053 bytes, lake-manifest `42fcfacc…`, sidecar `21d6e9d0…`). This is the
actor/task runtime model (no `Henret.Native`, no C compiler) — the same runtime iotakt's
real event loop builds on — so the pin fixes the task-lifecycle semantics jemmet's handler
policy assumes: spawn/schedule/complete/cancel, with `completed`/`cancelled` proven
terminal (henret's matrix). Only the `JemmetHenret` library depends on henret; the jemmet
core stays dependency-free.

## Pin policy (RFC 001)
- Dependencies are **pinned git revisions, never vendored.** iotakt and henret are separate
  projects; jemmet references them by URL + revision and lets Lake fetch them. The
  committed `lake-manifest.json` is the lockfile — it records the exact resolved commit, so
  a clean `lake build` is reproducible without trusting tag mutability.
- iotakt's consumer surface (`recvAck`/`sendAck`/`enableWrite`/`disableWrite`/
  `closeConnection`/`runStepAuto`/`FdKey`) is effectively frozen; remaining iotakt work
  (kqueue, `recvInto`) is additive-only and cannot break jemmet.
- jemmet binds **byte-level only**; iotakt's `Http`/`RequestBody`/`Router` stand-ins are
  superseded by jemmet's own (proven) parser/router and are not built or imported.
- kroopt (TLS) enters at M3 (RFC 009), to be pinned `git … @ "v0.124.0"`; not a dependency yet.

## Locking the dependencies (one-time, after this conversion)
`lake-manifest.json` is generated by Lake resolving the requires against the real repos:

```
lake update iotakt henret   # resolves the tags to commits, writes the lockfile
git add lake-manifest.json && git commit -m "lock iotakt 0.14.6 + henret 0.34.4"
```

henret will lock to `ad0ceab4ebed2884c9165be44154dca2c1f4816f` (verifiable against the pin
above). iotakt `v0.14.6` locks to its release commit. Commit the resulting manifest so
every subsequent build is byte-reproducible.
